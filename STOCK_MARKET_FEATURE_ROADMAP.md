# Stock Market Simulation Feature Roadmap

이 문서는 stock 모의투자를 실제 주식시장에 가깝게 확장할 때 기준으로 삼는 기능 범위와 우선순위를 정리한다.

`stock-back-service`는 사용자 API와 원장 계약의 기준점이고, `stock-batch-service`는 시세 수집, 자동장, 체결, 정산을 수행한다. `stock-front-service`는 이 원장을 조회하고 주문/관리 UI를 제공한다.

## 현재 구현된 기준

- 주문 접수: 매수, 매도
- 주문 유형: 시장가, 지정가
- 주문 취소: 전체 취소
- 시장 구분: `VIRTUAL_PRICE`, `ORDER_BOOK`
- 현재가 기준 체결: `VIRTUAL_PRICE` 주문을 현재가로 체결
- 수요와 공급 주문장 체결: `ORDER_BOOK` 매수/매도 주문을 가격 우선, 시간 우선으로 매칭
- 자동장: 자동 참여자가 실제 `stock_order` 원장에 주문 생성
- 가격 데이터: `stock_price`, `stock_price_tick`, Redis 최신가 캐시
- 주문장 종목 생성: 관리자 API와 프론트 관리자 화면
- 기업 이벤트 일부:
  - 초기 발행
  - 유상증자: 이벤트 등록, 권리락 가격 조정, 납입 상태 전이, 신주상장일 주식 수 반영
  - 추가 발행: 이벤트 등록, 신주상장일 주식 수 반영
  - 액면분할: 이벤트 등록, 효력일 주식 수/보유수량/가격 조정
  - 현금배당: 이벤트 등록, 배당락 가격 조정, 보유자별 지급 원장 생성, 지급일 현금 반영
  - 무상증자: 이벤트 등록, 권리락 가격 조정, 보유자별 신주 원장 생성, 신주상장일 보유수량/평균단가 반영
  - 주식배당: 이벤트 등록, 권리락 가격 조정, 보유자별 신주 원장 생성, 신주상장일 보유수량/평균단가 반영
- 계좌/보유/체결/정산:
  - 가상 계좌
  - 보유 수량
  - 예약 수량
  - 체결 이력
  - 장 마감 자산 스냅샷

## 1. 주문 관련 기능

현재는 시장가, 지정가, 취소 중심이다. 실제 시장에 가깝게 만들려면 아래 기능을 단계적으로 추가한다.

- 주문 정정: 기존 미체결 주문의 가격 또는 수량 변경
- 부분 취소: 주문 전체가 아니라 일부 잔량만 취소
- 조건부 주문:
  - stop
  - stop-limit
- 시간 조건:
  - 당일 주문
  - 즉시체결잔량취소 `IOC`
  - 전량체결아니면취소 `FOK`
- 장 시작/마감 주문:
  - 시가 주문
  - 종가 주문
- 주문 우선순위:
  - 가격 우선
  - 시간 우선
- 호가 단위:
  - 1원
  - 5원
  - 10원
  - 100원
  - 종목 가격대별 tick size

설계 기준:

- 주문 접수 API는 주문 원장을 만들고 예약금/예약수량을 관리한다.
- 체결 판단은 컨트롤러가 아니라 batch 또는 이후 분리될 trade engine에서 수행한다.
- 주문 정정은 기존 주문을 덮어쓰는 방식보다, 예약금/예약수량 재계산과 감사 추적이 가능하도록 별도 정정 이력 또는 버전 관리를 검토한다.

## 2. 시장 운영 흐름

실제 장은 하루 종일 같은 방식으로 체결되지 않는다. 시장 상태를 명시적으로 관리해야 한다.

- 장전 동시호가
- 시초가 결정
- 정규장 연속매매
- 변동성 완화장치
- 장중 거래정지
- 장마감 동시호가
- 종가 결정
- 시간외 거래
- 일일 가격제한폭

설계 기준:

- 종목 또는 시장 단위의 상태 테이블이 필요하다.
- 주문 접수 가능 여부와 체결 가능 여부는 별도로 판단한다.
- 거래정지 상태에서는 신규 주문, 정정, 취소 허용 범위를 명확히 분리한다.
- 가격제한폭은 전일종가 또는 기준가를 기준으로 계산한다.

## 3. 가격 형성 기능

수요와 공급 체결은 기본 매칭이다. 실제처럼 보이려면 체결 이후 가격 지표를 만들어야 한다.

- 최우선 매수호가
- 최우선 매도호가
- 체결가 결정 규칙
- 부분체결
- 마지막 체결가
- 기준가
- 전일종가
- 시가, 고가, 저가, 종가 `OHLC`
- 거래량
- 거래대금
- VWAP
- 등락률
- 상한가
- 하한가
- 호가 잔량
- 체결강도

설계 기준:

- `stock_execution`은 체결 원장이다.
- `stock_price`는 현재 조회용 최신 상태다.
- `stock_price_tick` 또는 별도 candle 테이블은 가격 이력과 차트용 데이터다.
- 거래량, 거래대금, OHLC는 체결 원장에서 집계하거나 배치로 materialize한다.

## 4. 기업 이벤트

현재 일부만 반영되어 있다.

초기 프로젝트에서 기업 이벤트를 더 늘릴지는 `docs/market-simulation/15-corporate-action-scope.md`를 기준으로 판단한다. 실제 시장에는 이벤트가 많지만, 현재 단계의 목표는 모든 이벤트 구현이 아니라 필수 원장 흐름을 안정화하는 것이다.

현재 반영:

- 초기 발행
- 유상증자
  - 발행가, 신주 수, 권리락일, 납입일, 신주상장일 등록
  - 권리락일에 이론권리락가격으로 기준 가격 조정
  - 납입일에 상태 전이
  - 신주상장일에 `issued_shares`, `tradable_shares` 증가
- 추가 발행
  - 발행가, 신주 수, 신주상장일 등록
  - 신주상장일에 `issued_shares`, `tradable_shares` 증가
- 액면분할
  - 분할 비율과 효력일 등록
  - 효력일에 `issued_shares`, `tradable_shares`, 보유수량, 예약수량을 같은 배율로 증가
  - 효력일에 현재가, 전일종가, 평균단가를 같은 배율로 나누어 조정
- 현금배당
  - 1주당 배당금, 배당락일, 지급일 등록
  - 배당락일에 배당락 가격으로 기준 가격 조정
  - 배당락일 현재 보유수량 기준으로 보유자별 지급 원장 생성
  - 지급일에 `stock_account.cash_balance` 증가

초기 필수 범위에서 제외하고 후속 검토로만 남길 기능:

- 감자
- 액면병합
- 신주인수권
- 권리공모
- 특별배당
- 자사주 매입/처분
- 주식교환
- 합병
- 분할
- 상장폐지
- 종목명/코드 변경

이 목록은 구현 예정 목록이 아니다. 각 항목은 단주, 권리, 종목 전환, 열린 주문, 가격 표시, 보유 가치 처리 같은 정책이 먼저 정해져야 하므로 초기 프로젝트에서는 enum, DDL, 프론트 선택지에 추가하지 않는다. 거래정지/거래재개는 기업 이벤트가 아니라 시장 운영 상태이므로 `MarketSessionStatus.HALTED` 경로에서만 다룬다.

설계 기준:

- 기업 이벤트는 주문 체결 로직과 분리된 원장 이벤트다.
- 이벤트 적용 전 열린 주문이 있으면 예약금, 예약수량, 가격 기준이 꼬일 수 있으므로 우선 거부한다.
- 액면분할과 액면병합은 보유수량과 평균단가를 같은 비율로 조정한다.
- 유상증자는 공시/권리락/납입/신주상장 단계를 분리하고, 신주상장일에 `issued_shares`, `tradable_shares`를 증가시킨다.
- 추가 발행은 별도 권리락 가격 조정 없이 신주상장일에 `issued_shares`, `tradable_shares`를 증가시킨다.
- 액면분할은 신주 발행처럼 회사 가치가 늘어나는 이벤트가 아니므로 주식 수와 가격을 반비례로 조정한다.
- 현금배당은 배당락 가격 조정과 지급일 현금 반영을 분리하고, 중복 지급을 막기 위해 보유자별 지급 원장을 둔다.
- 무상증자와 주식배당은 권리락 가격 조정과 보유자별 신주 entitlement를 분리한다. 현재 단주는 정수 주식만 배정하고 현금 보상은 하지 않는다.
- 감자/액면병합은 단주 처리 또는 현금 보상 정책이 먼저 정해져야 한다.

## 5. 결제/계좌 흐름

현재 프로젝트는 체결 즉시 현금과 보유수량을 반영하는 단순 모델에 가깝다. 실제 주식시장처럼 만들려면 체결과 결제를 분리해야 한다.

- 주문 접수
- 주문 체결
- 미수/예수금 반영
- 결제 예정
- 실제 결제
- 보유수량 확정
- 수수료
- 세금
- 실현손익
- 평균단가
- 평가손익

설계 기준:

- 단기 MVP에서는 즉시 결제 모델을 유지할 수 있다.
- 실제성 강화 단계에서는 `stock_settlement` 또는 유사한 결제 예정 원장이 필요하다.
- 수수료와 세금은 체결 단위로 계산하고, 계좌 잔액 반영 시 함께 차감한다.
- 실현손익은 매도 체결 시점에 계산하고, 평가손익은 현재가 기준으로 계산한다.
- 현재 구현은 batch 설정 기반 수수료/매도 거래세와 체결 단위 실현손익 기록까지 포함한다.

## 6. 리스크 / 시장 감시

모의투자라도 시장처럼 보이게 하려면 주문 제한과 감시 규칙이 필요하다.

- 주문 가능 금액 검증
- 보유수량 검증
- 이상 주문 제한
- 자기 주문 매칭 방지
- 너무 큰 주문 제한
- 가격 범위 제한
- 계좌별 주문 한도
- 종목별 거래정지
- 시장 전체 서킷브레이커

설계 기준:

- 사용자 입력 검증은 API에서 수행한다.
- 시장 상태, 가격 제한, 주문 한도는 도메인 서비스에서 공통 검증한다.
- 체결 엔진은 자기 주문 매칭 방지와 가격/시간 우선순위를 보장한다.
- 시장 전체 서킷브레이커는 모든 종목의 체결 가능 상태를 일괄 제어한다.

## 7. 투자자 행동

사용자가 할 수 있는 행동도 단계적으로 늘릴 수 있다.

- 매수
- 매도
- 정정
- 취소
- 보유 종목 조회
- 미체결 주문 조회
- 체결 내역 조회
- 손익 조회
- 관심종목
- 알림 설정
- 예약 주문
- 자동 손절/익절
- 배당 수령
- 유상증자 청약
- 신주인수권 행사
- 권리 이벤트 확인

설계 기준:

- 프론트는 브라우저 localStorage 기반 가짜 체결 상태를 만들지 않는다.
- 사용자의 행동은 API를 통해 DB 원장에 기록되어야 한다.
- 알림, 예약 주문, 자동 손절/익절은 별도 worker 또는 scheduler 책임으로 둔다.

## 추천 구현 우선순위

현재 프로젝트 기준으로는 아래 순서가 가장 현실적이다.

이미 구현된 기반:

- 주문 정정 / 부분 취소
- 호가 단위와 가격제한폭
- 종목별 장 상태: `OPEN`, `CLOSED`, `HALTED`
- 초기 필수 기업 이벤트: 초기 발행, 유상증자, 추가발행, 액면분할, 현금배당, 무상증자, 주식배당
- 체결 단위 비용/세금/실현손익

다음 우선순위:

1. 장 운영 일정: 장전 / 정규장 / 장마감 / 휴장
2. 시초가 / 종가 동시호가
3. 거래량, 거래대금, OHLC
4. 실현손익 기간별/종목별 집계 API
5. 기업 이벤트 필수 범위 안정화: 기준일, 단주 정책, 열린 주문 정책 테스트
6. 결제 예정 / 결제 완료 분리
7. 시장 감시 / 이상 주문 제한
8. 보류 이벤트 정책 문서화: 감자, 액면병합, 신주인수권, 권리공모는 구현 없이 필요한 원장 정책만 정리

## 다음 구현 후보

다음 작업은 `장 운영 일정 + 동시호가 + OHLC 집계`를 먼저 넣는 편이 좋다.

이유:

- 주문 정정/부분취소, 호가 단위, 가격제한폭, 장 상태는 이미 들어갔다.
- 초기 필수 기업 이벤트도 이미 충분히 들어갔으므로, 새 기업 이벤트를 더 늘리기보다 현재 원장 안정화를 먼저 해야 한다.
- 장 운영 일정이 있어야 장전 동시호가, 정규장, 거래정지, 장마감 정산을 자연스럽게 확장할 수 있다.
- OHLC와 거래량은 체결 원장을 사용자에게 시장처럼 보여주는 핵심 표시 기능이다.
